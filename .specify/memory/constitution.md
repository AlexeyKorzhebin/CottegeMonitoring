# CottageMonitoring Constitution

## Core Principles

### I. Двухуровневая архитектура

Система MUST быть разделена на две независимые части:

- **Облачная часть**: MQTT Broker, Python Daemon, PostgreSQL + TimescaleDB.
  Daemon читает значения с MQTT, сохраняет в БД, поддерживает актуальный
  срез значений объектов в реальном времени, предоставляет API.
- **Локальная часть**: контроллер LogicMachine, язык Lua, библиотека
  управления KNX-объектами.

Облачная и локальная части MUST быть развёрнуты, протестированы и обновлены
независимо друг от друга. Единственная точка связи — MQTT-брокер.

### II. MQTT как единая шина данных

Все потоки данных между контроллером и облаком MUST проходить через
MQTT-брокер:

- **Контроллер → Облако**: состояние устройств, данные KNX-объектов.
- **Облако → Контроллер**: команды управления (адрес объекта + значение).

Прямые соединения между локальной и облачной частями, минуя MQTT, запрещены.

### III. KNX-стандартизация объектов

Контроллер MUST приводить все объекты из разных протоколов к стандарту KNX
автоматически. `grp.all()` — единый API для получения списка объектов.
Формат данных объектов — JSON (см. `docs/objects.json`).

Любой новый протокол или тип устройства MUST быть доступен через единый
KNX-интерфейс контроллера без изменений в облачной части.

### IV. API-First

Облачная часть MUST предоставлять API для всех внешних потребителей:

- MCP-сервер
- Визуальные интерфейсы
- Сторонние интеграции

Актуальный срез значений объектов MUST быть доступен в реальном времени.
Данные временных рядов MUST быть доступны для аналитики и визуализации.

### V. Раздельное тестирование (NON-NEGOTIABLE)

На все функциональные и нефункциональные требования MUST быть разработаны
тесты. Тесты MUST быть разделены на два независимых набора:

- **Серверная часть** (Python): unit-тесты, integration-тесты, contract-тесты
  для API и MQTT-взаимодействия.
- **Контроллер** (Lua): тесты сервисов, тесты работы с KNX-объектами.

Каждый набор MUST запускаться и проходить независимо от другого.

### VI. Безопасность физического управления

Запись значения в KNX-объект означает управление реальным физическим
устройством. Все команды управления MUST быть валидированы перед отправкой
на контроллер. Некорректная команда может привести к повреждению
оборудования или угрозе безопасности.

MUST быть реализована проверка допустимых диапазонов значений и типов
данных для каждого объекта перед записью.

## Technology Stack

- **Облачная СУБД**: PostgreSQL с плагином TimescaleDB (временные ряды)
- **Облачный брокер**: MQTT Broker (облачное размещение)
- **Облачный сервис**: Python Daemon
- **Контроллер**: LogicMachine
- **Язык контроллера**: Lua
- **Библиотека контроллера**: встроенная библиотека KNX
  (https://kb.logicmachine.net/libraries/lua/)
- **Формат обмена данными**: JSON (MQTT-сообщения, API-ответы, объекты)
- **Справочники**:
  - Создание сервисов: https://kb.logicmachine.net/misc/apps/
  - Пример сервиса: `docs/samples/daemon.lua`
  - Пример выгрузки объектов: `docs/objects.json`

Изменение стека технологий требует внесения поправки в конституцию.

## Data Flow & Protocols

### От контроллера в облако

1. Контроллер собирает состояние всех устройств через KNX-интерфейс.
2. Данные публикуются в MQTT-брокер.
3. Python Daemon подписывается на топики, обрабатывает и сохраняет в
   PostgreSQL/TimescaleDB.
4. Daemon обновляет актуальный срез значений в реальном времени.

### От облака к контроллеру

1. Внешняя система формирует команду через API (адрес объекта + значение).
2. Python Daemon валидирует команду (Principle VI).
3. Команда публикуется в MQTT-брокер.
4. Контроллер получает команду и записывает значение в KNX-объект.

## Governance

Конституция является высшим документом проекта. Все технические решения,
спецификации и задачи MUST соответствовать изложенным принципам.

- **Поправки**: любое изменение конституции MUST быть задокументировано с
  указанием причины, одобрено и сопровождено планом миграции при нарушении
  обратной совместимости.
- **Версионирование**: MAJOR — удаление или несовместимое переопределение
  принципов; MINOR — добавление принципа или существенное расширение;
  PATCH — уточнения, исправления формулировок.
- **Ревью соответствия**: каждая спецификация и план реализации MUST
  содержать секцию Constitution Check, подтверждающую соответствие всем
  принципам.
