# Feature Specification: Server MQTT Ingestor

**Feature Branch**: `001-server-mqtt-ingestor`
**Created**: 2026-03-01
**Status**: Draft
**Input**: User description: "На основе docs/bt.md нужно разработать серверную часть. Доступ к серверу через ssh elion. Нужно реализовать протокол обмена, и сохранение в БД. А также получение команд. Реализовать тесты покрывающие все требования интеграции."

**Assumptions**:
- Сервер: `elion.black-castle.ru` (SSH-алиас: `elion`, ключи настроены, sudo доступен).
- Все сервисы (PostgreSQL, Redis, Mosquitto, nginx) установлены на elion или будут установлены при деплое.
- Приложение работает на elion как Docker-контейнер (`--network=host`), управляемый systemd — подключение к PostgreSQL, Redis, MQTT через `localhost`.
- Локальная разработка через SSH tunnel к elion (`ssh -L 5432:localhost:5432 -L 6379:localhost:6379 -L 1883:localhost:1883 elion -N`).
- Протокол обмена — MQTT v1 согласно `docs/bt.md` (namespace `{prefix}lm/<house_id>/v1/`).
- Dev/prod изоляция MQTT-топиков через `MQTT_TOPIC_PREFIX`: `dev/` для dev, пусто для prod.
- БД — PostgreSQL + TimescaleDB (согласно конституции проекта); две базы на одном сервере: `cottage_monitoring` (prod) и `cottage_monitoring_dev` (dev).
- На сервере установлена Grafana; дашборды — отдельная итерация.
- Frontend UI не входит в scope данной фичи (отдельная итерация).
  REST API входит в scope как требование Principle IV (API-First).
- Alert Engine не входит в scope данной фичи.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Приём и хранение телеметрии (Priority: P1)

Серверный сервис подключается к MQTT-брокеру, подписывается на топики
домов и непрерывно принимает данные: журнал событий (`events`), актуальное
состояние объектов (`state/ga/*`), метаданные схемы (`meta/objects` и чанки).
Все данные сохраняются в базу данных с разделением на текущий срез (state),
историю событий (events) и реестр схем (schema versions).

**Why this priority**: Без ingestion pipeline ни одна другая функция сервера
(команды, мониторинг, UI) невозможна. Это фундамент всей серверной части.

**Independent Test**: Подключить тестовый MQTT-клиент, опубликовать
сообщения в формате протокола v1 и убедиться, что данные корректно
сохранены в БД.

**Acceptance Scenarios**:

1. **Given** брокер работает и дом `house-01` публикует retained state,
   **When** ingestor запущен и подписан на `lm/house-01/v1/state/ga/+`,
   **Then** для каждого GA-объекта в БД создаётся/обновляется запись
   с `house_id`, `ga`, `ts`, `value`, `datatype`.

2. **Given** дом `house-01` публикует event в `lm/house-01/v1/events`,
   **When** ingestor получает сообщение,
   **Then** в таблице событий создаётся запись с полями `house_id`, `ts`,
   `seq`, `type`, `ga`, `value`, `datatype`, `raw_json`.

3. **Given** дом `house-01` публикует `meta/objects` с `schema_hash=H1`,
   **When** ingestor получает сообщение,
   **Then** создаётся запись в реестре схем с `house_id`, `schema_hash`,
   `ts`, `count`, `raw_meta_json`; объекты из meta сохраняются в таблицу
   objects.

4. **Given** meta опубликована чанками (`meta/objects/chunk/1..3`) с
   `schema_hash=H2` и `chunk_total=3`,
   **When** ingestor получает все 3 чанка,
   **Then** чанки собираются в полную схему и сохраняются как единая
   версия в реестре.

5. **Given** ingestor уже получил event с идентичным `(house_id, ts, ga, seq)`,
   **When** повторно приходит такой же event (QoS 1 дублирование),
   **Then** дубликат записывается в журнал как отдельная строка (append-only,
   дедупликация не выполняется).

### User Story 2 — Отправка команд и получение подтверждений (Priority: P2)

Серверный сервис формирует команды управления устройствами дома и
публикует их в MQTT. Контроллер выполняет команду и возвращает
подтверждение (ack). Сервер отслеживает связь между командой и
ack, сохраняет результат в БД.

**Why this priority**: Управление устройствами — ключевая ценность для
оператора. Без работающей цепочки команда → ack система является
read-only.

**Independent Test**: Отправить команду через сервис, симулировать ack от
тестового MQTT-клиента и проверить, что статус команды обновлён в БД.

**Acceptance Scenarios**:

1. **Given** сервис работает и дом `house-01` онлайн,
   **When** поступает запрос на отправку команды (ga=`1/1/1`, value=`1`),
   **Then** сервис генерирует `request_id`, публикует JSON в топик
   `lm/house-01/v1/cmd`, сохраняет команду в БД со статусом `sent`.

2. **Given** команда отправлена с `request_id=R1`,
   **When** контроллер публикует ack в `lm/house-01/v1/cmd/ack/R1`
   со `status=ok`,
   **Then** в БД обновляются поля `ts_ack`, `status=ok`, `results`.

3. **Given** команда отправлена с `request_id=R2`,
   **When** контроллер публикует ack со `status=error`,
   **Then** в БД сохраняется ошибка с описанием из `results`.

4. **Given** команда batch с несколькими GA-объектами,
   **When** отправляется batch-команда,
   **Then** все элементы передаются в одном MQTT-сообщении с единым
   `request_id`, ack содержит результат для каждого элемента.

5. **Given** команда с `request_id=R3` уже отправлена,
   **When** повторно отправляется cmd с тем же `request_id`,
   **Then** повторная бизнес-команда не создаётся (идемпотентность).

6. **Given** команда отправлена с `request_id=R4`,
   **When** ack не получен в течение 60 секунд,
   **Then** сервис выполняет retry — повторно публикует cmd с тем же
   `request_id`; количество retry-попыток ограничено (по умолчанию 2).
   Если после всех попыток ack не получен — статус команды обновляется
   на `timeout`.

7. **Given** команда в статусе `timeout` (retry исчерпаны),
   **When** ack всё же приходит позже (отложенный ответ),
   **Then** статус команды обновляется на `ok` или `error` из ack;
   факт позднего ответа логируется.

### User Story 3 — Мониторинг доступности и управление схемой (Priority: P3)

Серверный сервис отслеживает статус подключения каждого дома
(online/offline) через LWT-механизм MQTT и ведёт реестр домов.
При обнаружении нового `schema_hash` фиксирует новую версию схемы
объектов.

**Why this priority**: Знание о доступности дома и актуальной схеме
объектов необходимо для корректной работы оператора и инженера, но
может быть реализовано после базового ingestion и команд.

**Independent Test**: Симулировать online/offline через тестовый
MQTT-клиент (LWT) и проверить обновление статуса дома в БД.

**Acceptance Scenarios**:

1. **Given** дом `house-01` подключается к брокеру,
   **When** ingestor получает retained-сообщение
   `lm/house-01/v1/status/online` с `status=online`,
   **Then** в БД дом `house-01` создаётся (если не существует) со
   статусом `online` и обновляется `last_seen`.

2. **Given** дом `house-01` пропадает без корректного disconnect,
   **When** брокер публикует LWT с `status=offline`,
   **Then** в БД статус дома обновляется на `offline`.

3. **Given** ingestor получает `meta/objects` с новым `schema_hash`,
   **When** такой `schema_hash` ещё не зарегистрирован,
   **Then** создаётся новая версия схемы; предыдущая сохраняется
   для истории.

4. **Given** в предыдущей схеме (H1) было 50 объектов, а в новой (H2)
   появилось 3 новых объекта и отсутствует 2 старых,
   **When** ingestor принимает новую полную схему H2,
   **Then** 3 новых объекта добавляются в таблицу objects;
   2 отсутствующих помечаются неактивными (`is_active=false`) с привязкой
   к `schema_hash` последней схемы, в которой они присутствовали;
   физическое удаление записей НЕ происходит.

5. **Given** объект `1/1/5` помечен неактивным после смены схемы,
   **When** контроллер продолжает публиковать `state/ga/1/1/5` и events
   для этого GA (по старой логике или задержка обновления),
   **Then** state и events принимаются и сохраняются в БД без ошибки.

6. **Given** схема H1 содержит объект `{id:100, address:"1/1/1", name:"Свет"}`,
   а схема H2 содержит `{id:100, address:"1/1/1", name:"Свет крыльцо"}`,
   **When** ingestor принимает H2,
   **Then** запись в таблице objects обновляется: `name` меняется;
   предыдущая версия доступна через сохранённый `raw_meta_json` схемы H1.

7. **Given** дом `house-03` ранее не был известен системе,
   **When** ingestor получает первое сообщение (любого типа) с
   `house_id=house-03`,
   **Then** в БД автоматически создаётся запись дома с `house_id`,
   `created_at`, `online_status=unknown`; дальнейшие сообщения
   обрабатываются по общим правилам.

8. **Given** дом `house-02` выведен из эксплуатации (деактивирован
   оператором),
   **When** дом помечается `is_active=false`,
   **Then** все данные (state, events, objects, commands, schema)
   сохраняются в БД без изменений; ingestor прекращает обработку
   входящих сообщений для этого `house_id` и логирует их как
   предупреждение.

9. **Given** дом `house-02` был деактивирован (`is_active=false`),
   **When** оператор повторно активирует его (`is_active=true`),
   **Then** ingestor возобновляет обработку сообщений для этого
   `house_id`; накопленные retained-сообщения на брокере обрабатываются
   при следующей подписке/reconnect.

10. **Given** дом `house-01` находится в статусе `offline` более 30 дней
    и не был деактивирован,
    **When** дом не деактивирован явно,
    **Then** дом остаётся в реестре со статусом `offline`; автоматическое
    удаление или деактивация НЕ производится — решение принимает оператор.

11. **Given** дом публикует meta чанками, но не все чанки получены,
    **When** прошло допустимое время ожидания,
    **Then** сервер может запросить meta через RPC
    (`rpc/req/<client_id>` с `method=meta`).

### Edge Cases

- Что происходит при получении сообщения с невалидным JSON?
  Сообщение логируется как ошибка и пропускается; ingestion продолжается.
- Что если MQTT-брокер недоступен при старте сервиса?
  Сервис выполняет reconnect с экспоненциальной задержкой.
- Что если event приходит для неизвестного `house_id`?
  Дом автоматически регистрируется в БД с минимальными данными.
- Что если сообщение приходит для деактивированного дома (`is_active=false`)?
  Сообщение не обрабатывается, логируется как предупреждение.
  Данные не записываются в БД до повторной активации.
- Что если деактивированный дом снова появляется на брокере?
  Retained-сообщения остаются на брокере. При реактивации дома
  ingestor обработает retained state/meta при следующем подключении.
- Что если meta/objects содержит объекты с дублирующимся `id` или `address`?
  Последняя версия (по `schema_hash`) считается авторитетной; дубликаты
  внутри одной meta — ошибка, логируется.
- Что если event или state приходит для GA, отсутствующего в текущей схеме?
  Данные сохраняются без ошибки. Объект может быть ещё не зарегистрирован
  (meta не пришла) или уже помечен неактивным.
- Что если новая схема полностью пустая (0 объектов)?
  Все существующие объекты дома помечаются неактивными; версия
  сохраняется в реестре с `count=0`.
- Что если `ts` в сообщении сильно отличается от серверного времени?
  Сервер сохраняет `server_received_ts` наряду с payload `ts` для
  диагностики расхождений.

## Clarifications

### Session 2026-03-01

- Q: Что происходит, если ack на команду не приходит? → A: Таймаут 60 секунд. Перед финальным `timeout` выполняется retry (повторная отправка cmd). После исчерпания попыток — статус `timeout`.
- Q: Какой уровень наблюдаемости нужен серверному сервису? → A: Структурированные логи + метрики (ingest rate, lag, command latency, schema churn, online/offline). Формат метрик — Prometheus-совместимый для интеграции с Grafana (на сервере уже есть). Визуализация в Grafana — отдельная итерация.
- Q: Какая безопасность MQTT-подключения ingestor → брокер? → A: TLS + аутентификация логин/пароль. Credentials берутся из конфигурации/env. ACL на стороне брокера — отдельная задача.
- Q: Какова политика хранения событий (data retention)? → A: Без ограничений — хранить всё бессрочно. Ротация не реализуется на данном этапе.
- Q: Нужна ли дедупликация events? → A: Нет. Events — append-only журнал. QoS 1 дубликаты допустимы, unique constraint не требуется.

## Requirements *(mandatory)*

### Functional Requirements

**Протокол обмена (MQTT v1)**

- **FR-001**: Система MUST подписываться на топики по шаблону
  `{MQTT_TOPIC_PREFIX}lm/+/v1/#` для приёма данных от всех домов.
  Префикс задаётся через `MQTT_TOPIC_PREFIX` (пусто для prod, `dev/` для dev).
- **FR-002**: Система MUST разбирать топики по структуре
  `{MQTT_TOPIC_PREFIX}lm/<house_id>/v1/<subtopic>`: сначала отбрасывать
  настроенный `MQTT_TOPIC_PREFIX`, затем извлекать `house_id`.
- **FR-003**: Система MUST поддерживать все типы сообщений протокола v1:
  `events`, `state/ga/<ga>`, `meta/objects`, `meta/objects/chunk/<n>`,
  `status/online`, `cmd/ack/<request_id>`, `rpc/resp/<client_id>/<request_id>`.
- **FR-004**: Система MUST валидировать JSON-формат входящих сообщений
  и отклонять невалидные с записью ошибки в лог.

**Хранение состояния**

- **FR-005**: Система MUST хранить актуальный срез состояния (current state)
  каждого объекта по ключу `(house_id, ga)`.
- **FR-006**: При получении retained `state/ga/<ga>` система MUST выполнять
  upsert записи с полями: `house_id`, `ga`, `ts`, `value`, `datatype`.
- **FR-007**: Система MUST сохранять `server_received_ts` для каждой записи
  state наряду с payload `ts`.

**Хранение событий**

- **FR-008**: Система MUST записывать каждое event-сообщение в хранилище
  истории с полями: `house_id`, `ts`, `seq`, `type`, `ga`, `id`, `name`,
  `datatype`, `value`, `raw_json`.
- **FR-009**: Индексация событий MUST обеспечивать эффективные запросы
  по `(house_id, ts)` и `(house_id, ga, ts)`.
- **FR-048**: События MUST храниться бессрочно. Политика ротации или
  удаления старых данных на данном этапе НЕ реализуется.
- **FR-049**: Дедупликация events MUST NOT выполняться. Таблица событий
  работает в режиме append-only; дублирование записей при QoS 1
  допустимо. Unique constraint на события НЕ устанавливается.

**Схема объектов (Meta)**

- **FR-010**: Система MUST хранить версии схемы объектов (meta) с ключом
  `(house_id, schema_hash)`.
- **FR-011**: Система MUST собирать чанки meta по `(schema_hash, chunk_total)`.
  Когда все чанки получены — формировать полную схему.
- **FR-012**: Система MUST обновлять таблицу objects при получении новой
  полной схемы: `house_id`, `ga`, `object_id`, `name`, `datatype`,
  `units`, `tags`, `comment`, `schema_hash`.
- **FR-031**: При получении новой схемы система MUST сравнивать список
  объектов с предыдущей версией для того же `house_id` и определять:
  добавленные, удалённые, изменённые объекты.
- **FR-032**: Объекты, отсутствующие в новой схеме, MUST помечаться
  неактивными (`is_active=false`) в таблице objects. Физическое удаление
  записей запрещено — история state/events привязана к GA.
- **FR-033**: State и events для объектов, отсутствующих в текущей схеме
  (неактивных или ещё не зарегистрированных), MUST продолжать приниматься
  и сохраняться без ошибки.

**Дома и статус**

- **FR-013**: Система MUST вести реестр домов: `house_id`, `created_at`,
  `last_seen`, `online_status`.
- **FR-014**: При получении `status/online` с `status=online` система MUST
  обновить `online_status` дома и `last_seen`.
- **FR-015**: При получении LWT `status=offline` система MUST обновить
  `online_status` дома на `offline`.
- **FR-016**: Если `house_id` из входящего сообщения не найден в БД —
  система MUST автоматически создать запись дома с `is_active=true`.
- **FR-035**: Система MUST поддерживать деактивацию дома
  (`is_active=false`). При деактивации все данные дома (state, events,
  objects, commands, schema) MUST сохраняться в БД без изменений.
- **FR-036**: Для деактивированного дома (`is_active=false`) ingestor
  MUST прекращать обработку входящих MQTT-сообщений и логировать их
  как предупреждение.
- **FR-037**: Система MUST поддерживать реактивацию дома
  (`is_active=true`). После реактивации обработка сообщений
  возобновляется; retained-сообщения обрабатываются при следующем
  подключении/reconnect к брокеру.
- **FR-038**: Автоматическое удаление или деактивация дома по таймауту
  MUST NOT производиться. Решение о деактивации принимает оператор.

**Команды**

- **FR-017**: Система MUST предоставлять возможность отправки команды
  для дома: принять `(house_id, ga, value)` или batch
  `(house_id, items[{ga, value}])`.
- **FR-018**: Система MUST генерировать уникальный `request_id` (UUID) для
  каждой команды.
- **FR-019**: Система MUST публиковать команду в MQTT-топик
  `{MQTT_TOPIC_PREFIX}lm/<house_id>/v1/cmd` в JSON-формате протокола v1.
- **FR-020**: Система MUST сохранять команду в БД: `house_id`, `request_id`,
  `ts_sent`, `payload`.
- **FR-021**: При получении ack из `cmd/ack/<request_id>` система MUST
  обновить запись команды: `ts_ack`, `status`, `results`.
- **FR-022**: Повторная отправка cmd с тем же `request_id` MUST NOT
  создавать новую бизнес-команду (идемпотентность).
- **FR-040**: Если ack не получен в течение 60 секунд, система MUST
  выполнить retry — повторную публикацию cmd с тем же `request_id`.
  Максимальное количество retry-попыток: 2 (итого до 3 отправок).
- **FR-041**: После исчерпания всех retry-попыток без получения ack
  система MUST обновить статус команды на `timeout`.
- **FR-042**: Если ack приходит после установки статуса `timeout`,
  система MUST обновить статус команды из ack (`ok`/`error`) и
  зафиксировать факт позднего ответа в логе.

**RPC**

- **FR-023**: Система MUST предоставлять возможность запроса meta и
  snapshot через RPC: публикация в `rpc/req/<client_id>` и приём
  ответа из `rpc/resp/<client_id>/<request_id>`.

**Безопасность MQTT**

- **FR-046**: Система MUST поддерживать подключение к MQTT-брокеру по TLS
  (настраивается через `MQTT_USE_TLS`). Для localhost-подключений (сервисы
  на одном сервере) допускается работа без TLS.
- **FR-047**: Система MUST аутентифицироваться на MQTT-брокере с
  помощью логина/пароля. Credentials MUST читаться из конфигурации
  или переменных окружения; хранение credentials в коде запрещено.

**Надёжность**

- **FR-024**: Система MUST выполнять автоматический reconnect к
  MQTT-брокеру при обрыве соединения.
- **FR-025**: Система MUST обрабатывать burst-нагрузку без потери
  сообщений (внутренний буфер/очередь).

**Наблюдаемость**

- **FR-043**: Система MUST вести структурированные логи (JSON) с полями:
  `ts`, `level`, `house_id` (где применимо), `message_type`, `message`.
- **FR-044**: Система MUST экспортировать метрики в Prometheus-совместимом
  формате (endpoint `/metrics`) для интеграции с Grafana:
  - `ingestor_messages_total` — счётчик обработанных сообщений
    (labels: `house_id`, `message_type`)
  - `ingestor_lag_seconds` — разница `server_received_ts - payload.ts`
    (histogram, label: `house_id`)
  - `ingestor_house_status` — текущий статус дома: online/offline
    (gauge, label: `house_id`)
  - `ingestor_command_latency_seconds` — `ts_ack - ts_sent`
    (histogram, label: `house_id`)
  - `ingestor_command_timeout_total` — счётчик команд в статусе timeout
    (labels: `house_id`)
  - `ingestor_schema_changes_total` — счётчик смен schema_hash
    (label: `house_id`)
  - `ingestor_mqtt_reconnects_total` — счётчик переподключений к брокеру
- **FR-045**: Дашборды Grafana НЕ входят в scope данной фичи (отдельная
  итерация). Метрики MUST быть готовы к подключению без доработок
  серверного сервиса.

**Интеграционные тесты**

- **FR-026**: MUST быть реализованы интеграционные тесты для каждого
  типа сообщения протокола v1 (events, state, meta, meta/chunks,
  status, cmd, cmd/ack, rpc/req, rpc/resp).
- **FR-027**: Интеграционные тесты MUST проверять полный цикл:
  публикация MQTT-сообщения → обработка ingestor → проверка записи в БД.
- **FR-028**: Интеграционные тесты MUST проверять цикл команд:
  формирование команды → публикация в MQTT → симуляция ack → обновление
  статуса в БД.
- **FR-029**: Интеграционные тесты MUST проверять edge cases: невалидный
  JSON, дублирование QoS 1, неполные чанки meta, неизвестный `house_id`.
- **FR-030**: Интеграционные тесты MUST проверять reconnect к MQTT-брокеру
  после обрыва соединения.
- **FR-034**: Интеграционные тесты MUST проверять сценарии изменения
  схемы объектов: добавление новых объектов, удаление (soft delete)
  существующих, изменение атрибутов, приём state/events для неактивных
  объектов, пустая схема.
- **FR-039**: Интеграционные тесты MUST проверять жизненный цикл домов:
  автосоздание при первом сообщении, деактивация (игнорирование
  последующих сообщений), реактивация (возобновление обработки),
  сохранность данных деактивированного дома.

### Key Entities

- **House**: Дом/объект мониторинга. Атрибуты: идентификатор дома,
  время создания, время последней активности, текущий статус подключения,
  признак активности (`is_active`).
- **Object**: Объект автоматизации (KNX GA). Атрибуты: идентификатор дома,
  адрес GA, внутренний ID, название, тип данных, единицы измерения,
  теги, комментарий, хеш схемы, признак активности (`is_active`).
- **State**: Текущее состояние объекта. Атрибуты: идентификатор дома,
  адрес GA, метка времени, значение, тип данных.
- **Event**: Запись журнала событий. Атрибуты: идентификатор дома, метка
  времени, порядковый номер, тип события, адрес GA, значение, тип данных,
  исходный JSON.
- **SchemaVersion**: Версия схемы объектов. Атрибуты: идентификатор дома,
  хеш схемы, метка времени, количество объектов, исходный JSON.
- **Command**: Команда управления. Атрибуты: идентификатор дома,
  идентификатор запроса, время отправки, payload, время ack, статус,
  результаты.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Система обрабатывает поток событий от 10 домов одновременно
  без потери сообщений и без задержки обработки более 5 секунд.
- **SC-002**: Текущее состояние любого объекта любого дома доступно для
  чтения из БД в течение 1 секунды после получения обновления.
- **SC-003**: Команда управления доставляется до контроллера в течение
  2 секунд с момента формирования на сервере.
- **SC-004**: Все 49 функциональных требований (FR-001 — FR-049)
  покрыты интеграционными тестами с положительным результатом.
- **SC-005**: При обрыве связи с MQTT-брокером сервис восстанавливает
  подключение автоматически в течение 30 секунд.
- **SC-006**: Статус дома (online/offline) в БД обновляется в течение
  5 секунд после изменения подключения контроллера.
