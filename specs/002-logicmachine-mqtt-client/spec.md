## Clarifications

### Session 2026-03-03

- Q: Как обрабатывать несколько обновлений state для одного GA в буфере? → A: Сохранять каждое обновление (FIFO); GA on→off→on = 3 записи в буфере
- Q: Что делать при нажатии «Выгрузить в MQTT», если MQTT недоступен? → A: Показать сообщение об ошибке («MQTT недоступен»), выгрузка не выполняется

- Q: client_id — уникален в системе или только в доме? → A: Глобально уникален; house_id + device_id достаточно
- Q: После сохранения конфигурации daemon перезапускается автоматически? → A: Да, форма инициирует рестарт через apps/request.lp после config-save

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Настройка приложения через веб-интерфейс (Priority: P1)

Инженер или интегратор подключает контроллер Logic Machine к облачному серверу мониторинга. Открывает приложение в Apps на Logic Machine: форма настроек (режим dev/prod, адрес MQTT-брокера, логин/пароль, идентификаторы), кнопки «Выгрузить в файл» и «Выгрузить в MQTT» для ручной синхронизации. Сохраняет конфигурацию — daemon начинает работу.

**Why this priority**: Без настройки приложение не сможет подключиться к серверу. Это первый шаг при развёртывании.

**Independent Test**: Открыть форму настроек приложения, ввести значения, нажать «Сохранить» — проверить, что данные сохраняются (config.get) и daemon перезапускается с новыми настройками.

**Acceptance Scenarios**:

1. **Given** приложение установлено, **When** пользователь открывает настройки приложения, **Then** отображается форма с полями настроек и текущими значениями (config-load)
2. **Given** пользователь заполнил обязательные поля и нажал «Сохранить», **When** данные валидны, **Then** конфигурация сохраняется (config-save), окно закрывается и daemon автоматически перезапускается (через apps/request.lp)
3. **Given** пользователь ввёл невалидные данные, **When** нажимает «Сохранить», **Then** отображается сообщение об ошибке, конфигурация не сохраняется
4. **Given** приложение открыто, **When** пользователь нажимает «Выгрузить все объекты в файл», **Then** скачивается файл (JSON) с полной схемой meta/objects и текущими значениями
5. **Given** приложение настроено и MQTT подключён, **When** пользователь нажимает «Выгрузить все объекты в MQTT», **Then** вручную публикуются meta/objects и snapshot (state по всем GA) в выбранную среду (env_mode)
6. **Given** MQTT недоступен или daemon не подключён, **When** пользователь нажимает «Выгрузить все объекты в MQTT», **Then** отображается сообщение об ошибке («MQTT недоступен»), выгрузка не выполняется

### User Story 2 — Публикация телеметрии в облако (Priority: P2)

Daemon приложения слушает изменения на локальной шине (localbus): groupwrite, storage. При изменениях объектов отправляет в MQTT: события (events), обновлённое состояние (state/ga/*), метаданные схемы (meta/objects). Сервер мониторинга получает данные и сохраняет их.

**Why this priority**: Это основная ценность — оперативная видимость состояния дома в облаке.

**Independent Test**: Симулировать groupwrite на контроллере, проверить публикацию в MQTT топики events и state/ga/<ga>. Проверить, что сервер (spec 001) получает и обрабатывает данные.

**Acceptance Scenarios**:

1. **Given** приложение подключено к MQTT и слушает localbus, **When** происходит groupwrite на любой объект, **Then** публикуется event в `<prefix>cm/<house_id>/<device_id>/v1/events` и retained state в `<prefix>cm/<house_id>/<device_id>/v1/state/ga/<ga>` (prefix = `dev/` в dev-режиме, пусто в prod)
2. **Given** приложение запущено, **When** при первом старте (см. «Поведение при старте») или при изменении схемы объектов, **Then** публикуется meta/objects (полностью или чанками) и snapshot (state по всем GA)
3. **Given** MQTT недоступен, **When** происходят groupwrite на шине, **Then** события и state помещаются в RAM-буфер; при восстановлении соединения буфер публикуется в порядке FIFO
4. **Given** приложение подключается к брокеру, **When** установлено соединение, **Then** публикуется status/online с retain; при отключении — LWT status/offline

---

### User Story 3 — Получение и выполнение команд из облака (Priority: P3)

Daemon подписывается на топик команд `<prefix>cm/<house_id>/<device_id>/v1/cmd` (prefix зависит от env_mode). При получении команды (GA + value) выполняет запись через grp.write() на локальную шину и публикует подтверждение (ack) в соответствующий топик cmd/ack.

**Why this priority**: Управляемость — возможность удалённо управлять устройствами (свет, отопление и т.п.).

**Independent Test**: Отправить команду через API сервера (POST /api/v1/houses/{house_id}/commands), проверить MQTT publish в cmd и получение cmd/ack на сервере. Проверить, что значение применилось на шине.

**Acceptance Scenarios**:

1. **Given** приложение подписано на cmd, **When** приходит одиночная команда {request_id, ga, value}, **Then** выполняется grp.write(ga, value) и публикуется ack со status=ok/error
2. **Given** приходит batch-команда с items, **When** все пункты обработаны, **Then** публикуется один ack с результатами по каждому GA
3. **Given** команда содержит неизвестный GA или невалидное значение, **When** приложение обрабатывает её, **Then** публикуется ack со status=error и описанием ошибки в results

---

### User Story 4 — Обработка RPC-запросов (meta, snapshot) (Priority: P4)

Сервер может запрашивать полную схему (meta) или снимок состояния (snapshot) через RPC. Daemon подписывается на `<prefix>cm/<house_id>/<device_id>/v1/rpc/req/<client_id>`, при получении запроса формирует ответ и публикует в rpc/resp (prefix зависит от env_mode).

**Why this priority**: Позволяет серверу актуализировать схему и состояние при первом подключении или после долгого offline.

**Independent Test**: Отправить RPC meta через API, проверить публикацию rpc/req и получение rpc/resp. Проверить, что схема на сервере обновлена.

**Acceptance Scenarios**:

1. **Given** приложение подписано на rpc/req, **When** приходит запрос method=meta, **Then** формируется meta/objects и публикуется в rpc/resp
2. **Given** приходит запрос method=snapshot, **When** приложение формирует снимок текущего состояния всех экспортируемых объектов, **Then** публикуется rpc/resp с массивом state (ga, value, datatype)

### Edge Cases

- Что происходит при длительном отключении MQTT-брокера? Каждое изменение (event + state) — отдельная запись в буфере (без дедупликации). При восстановлении — публикация в FIFO. При переполнении — отбрасываются старые записи.
- Что происходит при невалидном JSON в cmd? Публикуется ack со status=error, приложение не падает.
- Что происходит при отсутствии объектов на шине? meta/objects с пустым массивом; events/state не публикуются до появления объектов.
- Что происходит при перезагрузке контроллера? См. раздел «Поведение при старте».

## Поведение при старте и начальная синхронизация

При запуске daemon выполняет последовательность:

1. **Подключение к MQTT** — установить соединение с брокером (TLS + логин/пароль).
2. **status/online** — опубликовать retain-сообщение о выходе онлайн (LWT настроить на status/offline).
3. **Подписка на cmd и rpc/req** — подготовить приём команд и RPC.
4. **meta/objects** — собрать схему всех объектов (grp.all) и опубликовать: если объектов ≤ 100 — одним сообщением в meta/objects; если > 100 — чанками в meta/objects/chunk/N.
5. **snapshot** — прочитать текущее значение каждого объекта и опубликовать retained state в state/ga/\<ga\> для всех объектов. Это даёт серверу полное состояние сразу после подключения.
6. **localbus listener** — начать прослушивание groupwrite и далее публиковать events и обновлять state при изменениях.

**Начальная синхронизация** завершена, когда meta и snapshot отправлены. Сервер получает схему и актуальное состояние и может показывать их в UI до первых изменений на шине.

**Ручной режим**: кнопка «Выгрузить все объекты в MQTT» в веб-интерфейсе повторяет шаги 4–5 (meta + snapshot) по требованию пользователя — для повторной синхронизации или проверки.

**Буфер при недоступности сети**: если MQTT отключён, каждое изменение (event + соответствующий state) помещается в RAM-буфер как отдельная запись (без дедупликации по GA). При восстановлении соединения публикуется содержимое буфера в порядке FIFO, затем — обычный поток. При `buffer_size=0` буфер отключён.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Приложение MUST быть реализовано как Logic Machine App: daemon (daemon.lua) + веб-интерфейс (index.lp) + конфигурация (config.lp или config.html) в соответствии с docs/apps.pdf и https://kb.logicmachine.net/misc/apps/
- **FR-002**: Daemon MUST использовать localbus для прослушивания groupwrite (и опционально groupread/groupresponse) и storage
- **FR-003**: Daemon MUST публиковать в MQTT топики согласно протоколу specs/001-server-mqtt-ingestor (contracts/mqtt-topics.md, docs/bt.md): events, state/ga/+, meta/objects, status/online. При числе объектов > 100 meta MUST публиковаться чанками (meta/objects/chunk/N)
- **FR-004**: Daemon MUST подписываться на cmd и rpc/req, публиковать cmd/ack и rpc/resp
- **FR-005**: Конфигурация MUST храниться через config.get/set, форма настроек MUST использовать события config-load, config-check, config-save. После успешного config-save форма MUST инициировать автоматический рестарт daemon (через apps/request.lp?action=restart&name=<appname>)
- **FR-006**: Приложение MUST поддерживать reconnect при обрыве соединения с MQTT-брокером
- **FR-006a**: При недоступности MQTT события (events) и обновления state MUST буферизоваться в RAM. Каждое изменение (event + state) — отдельная запись в буфере, без дедупликации по GA. Размер буфера задаётся настройкой `buffer_size`. При переполнении — отбрасываются старые записи (FIFO). При восстановлении соединения буфер MUST быть опубликован в порядке FIFO перед возобновлением обычной публикации
- **FR-007**: Экспорт MUST включать все объекты KNX-шины (без фильтрации по тегам)
- **FR-008**: При выполнении cmd MUST использоваться grp.write() для записи в локальную шину KNX
- **FR-009**: LWT (Last Will and Testament) MUST публиковать status/offline при аварийном отключении
- **FR-010**: Подключение к MQTT MUST использовать TLS и аутентификацию по логину/паролю. Используется единый логин/пароль для всего брокера (все дома — одни учётные данные)
- **FR-011**: Настройка `env_mode` MUST определять префикс топиков: `dev` → `dev/`, `prod` → пусто (см. contracts/mqtt-topics.md)
- **FR-012**: Опциональные настройки `snapshot_interval` и `throttle` MUST влиять на периодическую публикацию snapshot и ограничение частоты events соответственно
- **FR-013**: Веб-интерфейс приложения MUST содержать кнопку «Выгрузить все объекты в файл» — по нажатию формируется и скачивается JSON-файл с meta/objects и текущими значениями всех объектов
- **FR-014**: Веб-интерфейс MUST содержать кнопку «Выгрузить все объекты в MQTT» — по нажатию выполняется ручная публикация meta/objects и snapshot (state по всем GA) в MQTT согласно env_mode. Если MQTT недоступен или daemon не подключён — показать сообщение об ошибке, выгрузка не выполняется
- **FR-015**: При старте daemon MUST выполнять начальную синхронизацию: после подключения к MQTT — publish meta/objects и snapshot (state для всех объектов), затем запуск localbus listener

### Key Entities

- **House**: Идентификатор дома в системе мониторинга (house_id). Уникален, задаётся в настройках.
- **Device**: Идентификатор контроллера в рамках дома (device_id). Уникален в доме, задаётся в настройках. В паре с house_id даёт глобально уникальный client_id для MQTT.
- **Object**: KNX-объект (group address) — адрес, имя, тип данных, теги. Источник телеметрии и цель команд.
- **Configuration**: Настройки приложения (broker, house_id, device_id, фильтры и т.д.), хранятся через config API.

## Настройки веб-интерфейса (согласовано)

### Обязательные настройки

| Ключ | Описание | Пример | Валидация |
|------|----------|--------|-----------|
| `house_id` | Идентификатор дома в системе мониторинга | `house-01`, `dacha` | Не пусто, до 64 символов, буквы/цифры/дефис/подчёркивание |
| `device_id` | Идентификатор контроллера в рамках дома | `lm-main`, `lm-floor1` | Не пусто, до 64 символов |
| `env_mode` | Режим среды: куда отправлять телеметрию | `dev`, `prod` | `dev` = префикс `dev/` в топиках, `prod` = без префикса |
| `mqtt_host` | Адрес MQTT-брокера (хост или URL) | `elion.black-castle.ru`, `monitoring-dev.black-castle.ru` | Не пусто, домен или IP |
| `mqtt_port` | Порт MQTT-брокера | `8883` | Число 1–65535 |
| `mqtt_username` | Логин для MQTT | `cottage_client` | Не пусто |
| `mqtt_password` | Пароль для MQTT | `***` | Не пусто |
| `mqtt_use_tls` | Использовать TLS | `true` | Всегда true (обязательно) |

### Опциональные настройки

| Ключ | Описание | Пример | Валидация |
|------|----------|--------|-----------|
| `client_id` | MQTT Client ID (глобально уникален) | `lm-house01-main` | Строка или пусто (авто: house_id + device_id) |
| `debug` | Режим отладки (доп. логирование) | `true`, `false` | Булево, по умолчанию false |
| `snapshot_interval` | Интервал периодической публикации snapshot (секунды), 0 = отключено | `0`, `300`, `600` | Число ≥ 0, 0 = только по запросу RPC |
| `throttle` | Макс. частота events (событий в секунду) | `10`, `50`, `0` | Число ≥ 0, 0 = без ограничения |
| `buffer_size` | Размер RAM-буфера сообщений при недоступности MQTT (кол-во записей) | `500`, `1000`, `2000` | Число ≥ 0, 0 = буфер отключён; по умолчанию 1000 |

### MQTT-учётные данные: процесс получения

Используется **единый логин/пароль для всего MQTT-брокера** — все дома подключаются с одними учётными данными.

**Процесс:**

1. **На сервере (брокер)**: администратор создаёт пользователя MQTT (например, через `mosquitto_passwd` или конфиг Mosquitto) и задаёт логин с паролем.
2. **Передача инженеру**: логин и пароль передаются инженеру/интегратору вне приложения (документация, безопасный канал).
3. **В клиентском приложении**: инженер вводит `mqtt_username` и `mqtt_password` в форму настроек приложения на Logic Machine.
4. **Сохранение**: данные хранятся в config приложения (config.get/set). Пароль не покидает контроллер.

Сервер мониторинга (spec 001) использует свои собственные учётные данные для подписки на MQTT; учётные данные клиентов ему не требуются.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Инженер может настроить и запустить приложение на Logic Machine за менее чем 5 минут (ввод 8 обязательных полей + сохранение)
- **SC-002**: События groupwrite публикуются в облако в течение 2 секунд с момента изменения на шине
- **SC-003**: Команда из облака выполняется на контроллере (изменение на шине) в течение 3 секунд с момента отправки
- **SC-004**: При временном отключении брокера (до 5 минут) приложение восстанавливает соединение, публикует содержимое буфера и продолжает работу без ручного перезапуска
- **SC-005**: Контроллер с ~150 объектами публикует meta/objects при старте в течение 30 секунд
